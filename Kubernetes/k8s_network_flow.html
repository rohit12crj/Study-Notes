Kubernetes — FAANG interview flows

The 10 most common K8s flow questions asked at Google, Meta, Amazon, Apple, Netflix & Microsoft

1. What happens when you run kubectl apply?
kubectl→ API server→ AuthN + AuthZ→ Admission controllers→ etcd (persisted)→ Controller manager→ Scheduler→ kubelet→ Container runtime→ Pod Running
Key insight: apiserver is the only writer to etcd. Scheduler only assigns a node — kubelet actually starts the container.
2. How does an external HTTP request reach a Pod? (EKS)
Client→ Route 53→ ALB→ Ingress controller→ Service (ClusterIP)→ kube-proxy (iptables)→ Pod (VPC IP)
EKS uses AWS VPC CNI — Pods get real VPC IPs. ALB is provisioned by the AWS Load Balancer Controller via IRSA.
3. How does a Pod get scheduled onto a node?
Pod created (Pending)→ Scheduler: Filter→ Scheduler: Score→ Bind to node→ kubelet pulls spec→ Pull image→ Pod Running
Filter removes ineligible nodes (taints, resources, affinity). Score ranks remaining. On EKS, Karpenter provisions a new node if none fit.
4. How does a Pod autoscale? (HPA flow)
kubelet→ Metrics server→ HPA controller→ CPU > threshold→ Scale Deployment→ ReplicaSet creates Pods→ Karpenter adds node
HPA polls Metrics Server every 15s. On EKS, if new Pods are Pending due to no capacity, Karpenter provisions the right EC2 instance type automatically.
5. What happens when a Pod crashes / fails liveness probe?
Liveness probe fails→ kubelet detects→ Container killed→ RestartPolicy (Always)→ Restart w/ backoff→ CrashLoopBackOff
Readiness probe failure removes Pod from Service endpoints (no traffic) but does NOT restart it. Liveness failure restarts. Startup probe gives slow apps grace time.
6. How does a Pod access AWS services securely? (IRSA flow)
Pod (ServiceAccount)→ OIDC token mounted→ AWS STS→ IAM role assumed→ Temp credentials→ S3 / DynamoDB / etc.
No long-lived access keys ever stored in Pods. IRSA uses EKS OIDC provider + IAM trust policy. Each ServiceAccount maps to one IAM role — scoped per namespace.
7. How does a rolling deployment happen with zero downtime?
kubectl apply→ New ReplicaSet created→ New Pods start (maxSurge)→ Readiness probe passes→ Old Pods terminated (maxUnavailable)→ Rollout complete
maxSurge=1 means 1 extra Pod spun up. maxUnavailable=0 means no Pods removed until new one is Ready. PreStop hook + terminationGracePeriod drain in-flight requests.
8. How does a Secret get into a Pod securely? (EKS)
AWS Secrets Manager→ External Secrets Operator→ k8s Secret (etcd)→ KMS encrypted at rest→ Mounted as volume / env var→ Pod reads secret
Never hardcode secrets. Enable EKS envelope encryption with KMS. External Secrets Operator syncs from Secrets Manager using IRSA — no static credentials needed.
9. How does service-to-service communication work inside the cluster?
Pod A→ CoreDNS lookup→ ClusterIP resolved→ kube-proxy (NAT rule)→ Random endpoint selected→ Pod B
DNS format: my-svc.my-namespace.svc.cluster.local. kube-proxy translates ClusterIP → Pod IP via iptables DNAT. With a service mesh (Istio), the sidecar intercepts and adds mTLS.
10. What happens when a node goes down?
Node unreachable→ Node controller detects→ Node: NotReady (40s)→ Pods evicted (5min)→ Scheduler reschedules→ Karpenter replaces node→ Pods restored
Default eviction timeout is 5 minutes. PodDisruptionBudget ensures minimum replicas stay up during voluntary disruptions. On EKS, Karpenter detects EC2 termination and provisions a replacement.
