Kubernetes + AWS EKS — Revision Sheet

All components, key details, and how each maps to AWS EKS

Control plane — the brainRuns on master node
ComponentRoleKey detailsAWS EKS implementation
kube-apiserverThe Gatekeeper — single REST entry pointAuthenticates, authorizes & validates every request. Horizontally scalable. All components talk only through here.Managed AWS runs & manages it. Exposed as private/public endpoint. Access via aws eks update-kubeconfig. IAM authenticates via aws-auth ConfigMap.
etcdThe Memory — distributed KV storeOnly apiserver reads/writes it. Uses Raft consensus. Must be backed up. Single source of truth.Fully managed AWS manages etcd — encrypted at rest with AWS KMS. No direct access. AWS handles backups, HA, and replication.
kube-schedulerThe Placement Engine — assigns PodsFilters by resource requests, affinity, taints/tolerations. Scores nodes and picks the best fit.Managed Runs in AWS-managed control plane. Karpenter enhances scheduling by provisioning right-sized nodes just-in-time.
kube-controller-managerThe Reconciler — runs core control loopsNode, ReplicaSet, Endpoints, SA controllers. Each loop: observe → compare → act.Managed Fully managed by AWS. AWS CCM handles AWS-specific controllers (node lifecycle tied to EC2 instances).
cloud-controller-managerCloud Bridge — links k8s to cloud APIsManages cloud LBs, node lifecycle, routes. Absent in on-prem clusters.Managed AWS CCM built-in. Integrates with ELB for Services, EC2 for node registration, VPC for routes.
Data plane — worker nodesRuns on every node
ComponentRoleKey detailsAWS EKS implementation
kubeletThe Executor — primary node agentReads PodSpecs from apiserver. Starts containers via CRI. Does liveness/readiness probes. Reports status.Self-managed Runs on EC2 worker nodes. Pre-installed in EKS-optimized AMIs (Amazon Linux 2, Bottlerocket).
kube-proxyThe Networker — maintains Service routingUses iptables/IPVS for ClusterIP → Pod IP translation. Deployed as a DaemonSet.Managed add-on Deployed via EKS managed add-on. AWS VPC CNI can replace some duties with native VPC routing.
Container runtimeThe Engine — runs actual containersImplements CRI interface. containerd is default since k8s 1.24. Manages image pull and container lifecycle.Included in AMI containerd is default in EKS-optimized AMIs. Bottlerocket is AWS's purpose-built container OS.
PodAtomic Unit — smallest deployable entityShares network namespace (same IP). Init containers run before main. Ephemeral — replaced not restarted.EC2 or Fargate On EC2: Pods share the node. On Fargate: each Pod gets an isolated micro-VM billed per vCPU/memory.
Node typesWorker machine optionsConditions: Ready, MemoryPressure, DiskPressure. Physical or VM with CPU/RAM capacity.3 options Managed Node Groups, Self-managed nodes, or AWS Fargate (serverless — no EC2 to manage).
Networking
ComponentRoleKey detailsAWS EKS implementation
Service — ClusterIPInternal virtual IP for PodsDefault type. Stable IP inside cluster only. Backed by kube-proxy iptables rules.Standard EKS assigns IPs from VPC CIDR via AWS VPC CNI — Pods get real VPC IPs, not overlays.
Service — NodePortExternal access via node portOpens port 30000–32767 on every node. Traffic: NodeIP:Port → Service → Pod.Avoid in prod Works but rarely used. Security groups must open NodePort range. Use LoadBalancer or Ingress instead.
Service — LoadBalancerCloud load balancer per serviceProvisions a cloud LB per Service. Best for production external exposure.AWS LB Controller Creates ALB (L7) or NLB (L4). Annotate with service.beta.kubernetes.io/aws-load-balancer-type.
IngressL7 HTTP/HTTPS routingRoutes by hostname & path. TLS termination. Needs a controller. One LB for many services.AWS ALB AWS LB Controller maps Ingress → ALB. Supports WAF, ACM certificates, target group binding.
Ingress controllerImplements Ingress rulesNGINX, Traefik, HAProxy. Runs as a Pod. Watches Ingress objects and programs itself.AWS LB Controller Installed via Helm. Uses IRSA for creating/managing ALB/NLB. Recommended over NGINX on EKS.
CNI pluginPod networking layerAssigns Pod IPs, enables cross-node communication. Calico adds NetworkPolicy. Cilium uses eBPF.AWS VPC CNI Default CNI in EKS. Assigns real VPC IPs to Pods. Enable ENABLE_PREFIX_DELEGATION for more IPs per node.
CoreDNSInternal cluster DNSResolves svc.namespace.svc.cluster.local → ClusterIP. Runs as Deployment.Managed add-on EKS installs & manages CoreDNS. Version managed separately. Scale replicas for high DNS load.
NetworkPolicyPod-level firewall rulesIngress/egress rules by label. Default: allow all. Deny-all + explicit allow is best practice.VPC CNI / Calico EKS 1.25+ supports NetworkPolicy via VPC CNI network policy controller. Calico/Cilium for richer policies.

Workloads
KindUse caseKey detailsAWS EKS implementation
DeploymentStateless apps + rolling updatesManages ReplicaSets. RollingUpdate & Recreate strategies. Easy rollback via revision history.Standard Use with HPA + Karpenter for auto-scaling. ECR for images. CodePipeline/ArgoCD for CI/CD.
ReplicaSetMaintains exact Pod countSelector-based Pod ownership. Usually managed by Deployment — rarely created directly.Standard Managed by Deployment. Use topologySpreadConstraints to spread replicas across AZs.
StatefulSetStateful apps — DBs, queuesStable Pod names (pod-0, pod-1), stable network IDs, ordered scale. Each Pod gets its own PVC.EBS / EFS Use EBS (gp3) for RWO. Use EFS (aws-efs-csi-driver) for RWX. EBS is AZ-bound — plan carefully.
DaemonSetOne Pod per nodeAuto-adds Pod on new nodes. Used for log collectors, Prometheus node-exporter, CNI plugins.Standard AWS VPC CNI, kube-proxy, CloudWatch agent run as DaemonSets. Fluent Bit ships logs to CloudWatch.
Job / CronJobBatch & scheduled tasksJob: runs to completion, retries on failure. CronJob: cron-scheduled. TTL for cleanup.Fargate-friendly CronJobs work well on Fargate (no idle node cost). AWS Batch for heavy batch workloads.
HPAHorizontal Pod autoscalingScales replicas on CPU/memory/custom metrics. Min/max bounds. Cooldown prevents thrashing.KEDA / CW Metrics Use KEDA with SQS queue depth, Kinesis lag, or CloudWatch metrics. Metrics Server required for CPU/mem.
Configuration & secrets
ResourcePurposeKey detailsAWS EKS implementation
ConfigMapNon-sensitive configurationKey-value pairs or files. Injected as env vars or volume mounts. Volume-mounted changes reflect without Pod restart.Standard Works natively. For dynamic config, use AWS AppConfig synced via the AppConfig agent sidecar.
SecretSensitive data storageBase64-encoded in etcd (not encrypted by default). Types: Opaque, TLS, dockerconfigjson.AWS Secrets Manager External Secrets Operator or Secrets Store CSI Driver syncs from Secrets Manager / SSM. etcd encrypted via KMS.
Resource requests & limitsCPU/memory bounds per containerRequests = scheduling guarantee. Limits = hard ceiling. CPU throttled, memory limit = OOMKilled.Critical for cost Right-sizing affects EC2/Fargate cost. Use VPA or Goldilocks for recommendations. Fargate charges per requested vCPU/mem.
ProbesContainer health checksLiveness: restart if failing. Readiness: remove from Service. Startup: grace for slow starts. HTTP/TCP/exec types.Standard ALB target group health checks complement readiness probes. Container Insights surfaces probe failure counts.
NamespacesLogical cluster isolationScope for RBAC, quotas, NetworkPolicies. Default: default, kube-system, kube-public.Multi-tenant pattern One namespace per team/env. IRSA scoped per namespace. ResourceQuota prevents noisy-neighbor on shared clusters.
IRSAIAM Roles for Service AccountsLinks k8s ServiceAccount to AWS IAM role via OIDC. Pods get scoped AWS credentials — no long-lived keys.EKS-native EKS OIDC provider + annotate SA with eks.amazonaws.com/role-arn. Best practice for any Pod needing AWS API access.
Storage
ResourceRoleKey detailsAWS EKS implementation
PersistentVolume (PV)Actual provisioned storageCluster-wide resource. Has capacity, accessMode, reclaimPolicy (Retain/Delete/Recycle).EBS / EFS / FSx Dynamically provisioned via CSI drivers. EBS PVs are AZ-locked — StatefulSet Pods must stay in same AZ.
PVCPod's request for storageBound to a matching PV. Lifecycle independent of Pod. Deleted separately unless reclaimPolicy=Delete.Standard Use volumeClaimTemplates in StatefulSet for per-Pod PVCs backed by EBS gp3.
StorageClassDynamic provisioning templateDefines provisioner, parameters (IOPS, type), reclaimPolicy. PVC references it to auto-create PV.EBS gp3 default EKS ships with gp2 default. Best practice: create gp3 StorageClass (cheaper, better IOPS). io2 for databases.
CSI driversStorage plugin interfaceVendor-neutral standard. Decouples storage plugins from k8s core. Installed separately.AWS CSI drivers aws-ebs-csi-driver (RWO), aws-efs-csi-driver (RWX), aws-fsx-csi-driver (Lustre). Install as EKS managed add-ons.
Access modesRWO / ROX / RWXReadWriteOnce (1 node), ReadOnlyMany, ReadWriteMany. Block storage (EBS) is RWO only.EFS for RWX EBS = RWO only. EFS = RWX — multiple Pods across nodes can mount same filesystem.
Security
ResourceRoleKey detailsAWS EKS implementation
RBACWho can do what in k8sRole + RoleBinding (namespace). ClusterRole + ClusterRoleBinding (cluster-wide). Least-privilege principle.aws-auth ConfigMap Maps IAM users/roles → k8s RBAC groups. EKS access entries (new API) replace aws-auth. IAM = authn, RBAC = authz.
ServiceAccount + IRSAPod AWS identityEvery Pod runs as a SA. Default SA has minimal permissions. Annotate SA to assume IAM role via OIDC.EKS OIDC Enable OIDC provider on EKS cluster. Annotate SA with IAM role ARN. No long-lived credentials in Pods.
Secrets encryptionProtect sensitive data at restBase64 in etcd by default — not secure. Enable EncryptionConfiguration. Use external secrets manager in prod.AWS KMS Enable EKS envelope encryption with CMK. Secrets Store CSI Driver or External Secrets Operator pulls from Secrets Manager / SSM.
NetworkPolicyPod traffic firewallIngress/egress rules by label. Default: allow all. Apply deny-all + explicit allow per microservice.VPC CNI / Calico EKS 1.25+ supports NetworkPolicy via VPC CNI network policy controller. Security Groups for Pods adds ENI-level SG rules.
Pod securityContainer security profilesPodSecurity admission (v1.25+): Privileged / Baseline / Restricted. Label namespace to enforce profile.OPA / Kyverno Use Kyverno or OPA Gatekeeper for fine-grained policies. GuardDuty EKS Protection detects runtime threats.
Image securityTrusted container images onlyOnly pull from trusted registries. Scan for CVEs. Enforce via admission controller.Amazon ECR ECR as private registry — integrated IAM auth. ECR image scanning (Inspector) on push. No credentials in Pods.
Scaling & observability
ToolRoleKey detailsAWS EKS implementation
HPAScale Pod replicas automaticallyScales on CPU/memory/custom metrics. Min/max bounds. Cooldown prevents thrashing.Metrics Server Install Metrics Server as EKS add-on. Use KEDA for SQS/Kinesis/CloudWatch-based scaling.
VPARight-size Pod resourcesAdjusts requests/limits. Off/Auto/Initial modes. Restarts Pods to apply changes.Goldilocks Install VPA + Goldilocks dashboard to visualize right-sizing recommendations without auto-applying.
Cluster autoscalerScale EC2 nodes up/downAdds nodes when Pods are Pending. Removes under-utilized nodes. Works with ASGs.Karpenter preferred Karpenter is faster, picks optimal EC2 types, supports Spot, and consolidates nodes. Preferred for EKS.
KEDAEvent-driven Pod scalingScales on queue depth, Kafka lag, cron, HTTP. Can scale to zero. Works alongside HPA.SQS / Kinesis KEDA SQS scaler scales consumers based on queue depth. Uses IRSA for AWS API access. Install via Helm.
Metrics serverCore metrics pipelineCPU/mem from kubelets. Feeds HPA & kubectl top. Not for long-term storage.EKS add-on Available as EKS managed add-on. Use Amazon Managed Prometheus (AMP) + Grafana (AMG) for full observability.
LoggingContainer & control plane logsApp logs to stdout/stderr. Ship to central store. Control plane logs separate from node logs.CloudWatch / S3 Enable EKS control plane logging to CloudWatch. Fluent Bit DaemonSet ships logs to CloudWatch Logs or OpenSearch.
TracingDistributed request tracingTrack requests across microservices. Identify latency bottlenecks. Requires instrumentation.AWS X-Ray ADOT (AWS Distro for OpenTelemetry) collector as sidecar/DaemonSet. Preferred approach over X-Ray SDK directly.
